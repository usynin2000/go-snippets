# 📘 Горутины и каналы в Go

---

## 1. Горутины — что это такое?

**Аналогия:** Представь ресторан.

- `main()` — это менеджер зала.
- Обычный вызов функции — менеджер сам идёт на кухню, стоит там и ждёт, пока приготовят блюдо.
- Горутина (`go someFunc()`) — менеджер говорит повару "приготовь", и идёт заниматься другим. Повар работает **параллельно**.

```go
// Менеджер стоит и ждёт — программа "зависает" на каждом вызове
slowFunction("Задача 1")
slowFunction("Задача 2")

// Менеджер раздаёт задачи и идёт дальше — всё работает параллельно
go slowFunction("Задача 1")
go slowFunction("Задача 2")
```

**Главная ловушка:** Если менеджер уйдёт домой (main завершится), все повара тоже уходят — горутины убиваются.

```go
go slowFunction("Задача") // горутина запущена
// НО если main завершится прямо сейчас — горутина не успеет отработать!
```

Чтобы этого не было, нужно как-то подождать горутины. Для этого и нужны каналы.

---

## 2. Каналы — что это такое?

**Аналогия:** Канал — это труба или окошко передачи между горутинами.

- Один повар (горутина) кладёт блюдо в окошко → `ch <- блюдо`
- Другой забирает → `блюдо := <-ch`
- **Пока никто не забрал — повар стоит у окошка и ждёт.** Это и есть блокировка.

```go
ch := make(chan string)

go func() {
    ch <- "Привет из горутины!" // отправляет и ЖДЁТ, пока кто-то прочитает
}()

msg := <-ch // читает — только тут горутина "разблокируется"
fmt.Println(msg)
```

**Почему это лучше, чем просто переменная?**

Если бы две горутины одновременно писали в одну переменную — был бы хаос (гонка данных).
Канал — это безопасная передача: один передаёт, другой получает, порядок гарантирован.

---

## 3. Небуферизованный vs Буферизованный канал

### Небуферизованный — "передача из рук в руки"

```go
ch := make(chan int)
```

Повар приготовил блюдо → держит его в руках → **ждёт, пока официант не подойдёт и не возьмёт**.
Только когда официант взял — повар может готовить следующее.

```go
go func() {
    ch <- 1  // стоит с блюдом в руках, ждёт официанта
    ch <- 2  // второе блюдо начнёт готовить только после того, как заберут первое
}()
```

### Буферизованный — "поднос на раздаче"

```go
ch := make(chan int, 3) // поднос на 3 блюда
```

Повар ставит блюда на поднос и **не ждёт официанта** — продолжает готовить.
Официант подходит и забирает, когда готов.
Если поднос полный — повар снова вынужден ждать.

```go
ch <- 1 // ставит на поднос, не ждёт
ch <- 2 // ставит на поднос, не ждёт
ch <- 3 // ставит на поднос, не ждёт
ch <- 4 // ПОДНОС ПОЛНЫЙ — ждёт, пока официант не заберёт хоть одно
```

**Итог:**
- Небуферизованный = синхронизация (оба должны быть готовы одновременно)
- Буферизованный = очередь (отправитель может забежать вперёд)

## 4. Закрытие канала

Когда повар закончил смену — он говорит: "Больше блюд не будет" → `close(ch)`

```go
go func() {
    ch <- 1
    ch <- 2
    close(ch) // сигнал: всё, данных больше не будет
}()

for val := range ch { // range сам остановится, когда канал закрыт и пуст
    fmt.Println(val)
}
```

**Проверить вручную:**
```go
val, ok := <-ch
if !ok {
    fmt.Println("Канал закрыт, данных нет")
}
```

**Правило:** Закрывает всегда тот, кто **отправляет**. Получатель никогда не закрывает.

## 5. select — ждём сразу несколько каналов

**Аналогия:** Ты сидишь в кафе и ждёшь либо звонка от друга, либо уведомления от такси — кто первый откликнется, того и слушаешь.

```go
select {
case msg := <-ch1:
    fmt.Println("Друг позвонил:", msg)
case msg := <-ch2:
    fmt.Println("Такси приехало:", msg)
}
```

**С таймаутом** — если никто не откликнулся за 2 секунды:
```go
select {
case msg := <-ch:
    fmt.Println("Получили:", msg)
case <-time.After(2 * time.Second):
    fmt.Println("Никто не ответил, идём без них")
}
```

**С `default`** — не блокироваться вообще:
```go
select {
case msg := <-ch:
    fmt.Println("Есть данные:", msg)
default:
    fmt.Println("Канал пуст — продолжаем дальше, не ждём")
}
```

## 6. Итог: как всё связано

```
goroutine 1 ──── ch <- данные ────▶  канал  ────▶  данные := <-ch ──── goroutine 2
                  (отправляет)      (труба)          (получает)
```

- **Горутина** — параллельный исполнитель
- **Канал** — безопасная труба для передачи данных между горутинами
- **Небуферизованный** — синхронизирует (оба ждут друг друга)
- **Буферизованный** — позволяет отправителю обгонять получателя
- **close** — сигнал "данных больше не будет"
- **select** — ждёт сразу несколько каналов, реагирует на первый готовый
